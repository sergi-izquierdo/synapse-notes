\section{UI optimista amb React 19 (useOptimistic + useTransition)}
\label{sec:optimistic-ui}
% Subsecció per al redactat final. Afegida 2026-04-24.
%
% Commit \texttt{e94aea5}. Les accions d'estrella, arxiu i
% eliminació d'una nota havien de passar per una server action
% + \texttt{revalidatePath} + RSC re-fetch. En producció això
% generava 1-2~s de latència percebuda: l'usuari clicava un
% botó, el botó feia feedback visual (\texttt{disabled},
% spinner), la xarxa anava al backend, el backend actualitzava
% la BD, retornava OK, i només aleshores la UI es re-renderitzava
% amb el nou estat. En dispositius mòbils o connexions lentes,
% la UI es notava ``lenta'' tot i que la lògica era correcta.
%
% \subsection{Patró aplicat}
%
% React 19 exposa dos hooks nous que resolen el problema sense
% trencar el flux server-first:
%
% \begin{itemize}
%   \item \texttt{useOptimistic(state, reducer)} --- defineix un
%     estat derivat que pot ser actualitzat instantàniament
%     abans del round-trip al servidor, i que es reverteix
%     automàticament quan el servidor retorna el ``veritable''
%     estat via \texttt{revalidatePath}.
%   \item \texttt{useTransition()} --- marca la mutation com a
%     ``low-priority'' perquè React la pugui interrompre si
%     l'usuari fa una altra acció abans del round-trip.
% \end{itemize}
%
% La implementació a \texttt{NoteGrid} defineix dues accions
% optimistes:
%
% \begin{itemize}
%   \item \texttt{\{ type: 'patch', id, patch \}} --- merge de
%     camps a la nota corresponent (p.ex. toggle de
%     \texttt{starred}).
%   \item \texttt{\{ type: 'remove', id \}} --- filtra la nota
%     fora del llistat visible (archive/delete).
% \end{itemize}
%
% Cada handler encaixa:
%
% \begin{verbatim}
% const [optimisticNotes, applyOptimistic] = useOptimistic(
%   notes,
%   (state, action) => { ... }
% );
% const [, startMutation] = useTransition();
%
% const handleStar = (id) => {
%   startMutation(async () => {
%     applyOptimistic({ type: 'patch', id,
%       patch: { starred: !current.starred } });
%     await toggleNoteStarred(id);
%     // revalidatePath inside the action refreshes
%     // the RSC props; React discards our optimistic
%     // state automatically.
%   });
% };
% \end{verbatim}
%
% \subsection{Què NO va a l'estat optimista}
%
% \texttt{createNote} i \texttt{duplicateNote} queden fora del
% patró, i és una decisió deliberada:
%
% \begin{itemize}
%   \item L'\textbf{id} d'una nota és server-assigned
%     (\texttt{bigserial}). Qualsevol estat optimista que
%     renderitzi la nova nota necessitaria inventar un id
%     temporal, que col·lisionaria amb la realitat quan la
%     \texttt{revalidatePath} retorni.
%   \item Generar l'\textbf{embedding} (768-dim, ~200~ms a
%     Gemini) és el coll d'ampolla real. Pintar la nota abans
%     que l'embedding estigui a la BD crearia un state on la
%     nota existeix al UI però no és buscable per RAG --- i
%     això sí que és visible per l'usuari (el xat no la
%     troba). Pintar l'spinner de ``creant...'' és la resposta
%     correcta.
% \end{itemize}
%
% La lliçó defensable per al tribunal: el patró optimista és
% \textbf{per defecte per a mutations ràpides amb
% revalidatePath}, però cal una excepció explícita quan el
% servidor assigna identificadors o el bottleneck real és la
% generació de dades derivades. Aquesta decisió quedà anotada
% a memòria persistent del workflow
% (\texttt{feedback\_optimistic\_ui\_default.md}).

%
% Abans de començar la Part~B (MCP), es va afegir una funcionalitat
% d'UX que té una decisió tècnica defensable davant el tribunal:
% reordenació manual de notes via drag-and-drop, amb ordre persistit
% a Postgres.
%
% Decisions tècniques destacables:
% 1. **Llibreria de drag:** `@dnd-kit` en comptes de
%    `react-beautiful-dnd` (deprecated) o Framer Motion `Reorder`
%    (només single-axis, no suporta grids). `@dnd-kit` és l'única
%    opció 2026-compatible amb React 19 i suport grid.
% 2. **Persistència:** fractional indexing (paquet
%    `fractional-indexing` de Rocicorp) en lloc de `position int`.
%    Raonament: `position int` requereix N+1 UPDATEs per cada
%    reordenació (el pattern anti-exemple de Figma, Notion, Linear),
%    mentre fractional indexing només requereix una UPDATE. Trade-off:
%    keys lexicogràfiques de llargada variable vs. precisió il·limitada
%    (floats s'esgoten després de ~52 midpoint-splits).
% 3. **Swap vs insert:** després de feedback d'acceptació, canviada
%    la semàntica de "insert with shift-left cascade" a "swap with
%    hover target". El hover target es mou a la posició d'origen,
%    la resta queda fixa. Més comprensible visualment.
% 4. **Mobile:** `TouchSensor` amb `delay: 200ms, tolerance: 15px`
%    perquè long-press activi el drag sense conflictar amb scroll.
%    `touch-action: none` descartat (bloqueja scroll fins i tot
%    durant la delay window).
%
% Bugs trobats i corregits (defensables com a "lessons learned"):
% - Fractional-indexing rebutja keys amb part fraccional acabada en
%   `'0'` (p.ex. `a000010`). El backfill inicial produïa keys
%   invàlides; corregit amb migració `20260425150000_notes_position_
%   canonical.sql` que reescriu totes les posicions a format
%   library-canonical `a0`..`a9`,`aA`..`aZ`,`aa`..`az`.
% - Framer Motion `layout` + variants amb `y:0` sobreescriuen el
%   transform de dnd-kit. Variants reduïts a opacity-only; layout
%   eliminat del SortableNoteCard.
% - Original note "clone" visible durant drag — resolt amb
%   `DragOverlay` portal + `invisible` / outlined placeholder al
%   slot d'origen.

\section{Física del graph: tethering estil Obsidian}
\label{sec:graph-physics}
% Subsecció per al redactat final. Afegida 2026-04-24.
%
% El visualitzador base (\S\ref{sec:neural-graph}) usa
% `react-force-graph-2d` amb els defaults de d3-force, que són
% calibrats per a grafs densos de 100+ nodes. Sobre la corpus
% típica d'un usuari de Synapse Notes (12-50 notes, sparse), el
% comportament era patològic:
%
% 1. Arrossegar un node pertorbava lleugerament la resta i la
%    repulsió propagava la pertorbació a tota la tela
%    (efecte "blast").
% 2. Nodes sense arestes (orphans — p.ex. una nota única amb tag
%    únic) derivaven cap al centre i s'apilaven visualment a
%    sobre dels nodes d'un clúster.
% 3. La càmera auto-fit feia zoom-out al perímetre expandit
%    temporalment i no tornava.
%
% El target de disseny és el graph view d'Obsidian: una massa
% central amb un perímetre virtual dins del qual les notes es
% mouen lliurement, i que retornen naturalment quan es
% desplacen fora. No un perímetre rígid (boundary check), sinó
% un equilibri físic entre tres forces:
%
% \begin{enumerate}
%   \item \textbf{charge (repulsió)}: strength $-55$ amb
%     \texttt{distanceMax = 500~px}. Multi-body Barnes-Hut
%     inverse-square, però acotada al rang visible per evitar que
%     un drag ondulés a través de tot el graf.
%   \item \textbf{gravetat (forceX/forceY)}: strength $0.07$ per
%     eix. Pull lineal per-node cap a l'origen; la força creix amb
%     la distància, de manera que un node proper sent poc però un
%     node al corner de la tela sent prou per tornar.
%   \item \textbf{collision (forceCollide)}: radi $38~\text{px}$,
%     strength $0.95$. Resolució geomètrica hard — tracta cada
%     node com un disc i impedeix el solapament directe (que
%     charge, a curt rang, no aconsegueix eficientment).
% \end{enumerate}
%
% L'equilibri entre charge i gravetat té forma $d^3 \propto
% |Q|/k_g$ amb $Q$ la càrrega i $k_g$ la strength de gravetat.
% Amb $|Q|=55$ i $k_g=0.07$: $d \approx 9.3$ en unitats de d3
% (≈~80-100~px a zoom 1×), que és el radi del clúster estable.
% El collision radius ($38~\text{px}$) és inferior al radi
% d'equilibri però superior a les dimensions del node dibuixat
% (4-6~px) + label renderitzat (~35~px vertical), impedint
% colisions visuals sense dominar la física.
%
% \textbf{Bug operacional trobat}: el ref imperativa a
% ForceGraph2D (\texttt{useRef}) no es capturava perquè el
% component estava dinàmicament importat amb
% \texttt{next/dynamic} (\texttt{ssr: false}). L'efecte
% \texttt{useEffect} disparava quan arribaven les dades, veia
% \texttt{ref.current === null} perquè el mòdul encara no
% s'havia carregat, i feia no-op silenciós. Cap tuning
% s'aplicava mai. Solució: ref basada en \texttt{useState}
% passada com a callback-ref. Aquest patró queda documentat
% com a lliçó per a qualsevol integració futura amb
% \texttt{next/dynamic}.

%
% Feature que combina dues peces ja presents al sistema: el model
% Claude Haiku 4.5 (usat al xat principal) i el patró de
% structured output del Vercel AI SDK. Donada la composició d'una
% nota (contingut + llista d'etiquetes existents del propi
% usuari), l'API retorna fins a 3 etiquetes existents aplicables
% i, opcionalment, una proposta d'etiqueta nova normalitzada.
%
% \subsection{Arquitectura}
%
% - \textbf{Ruta}: \texttt{POST /api/suggest-tags} --- auth-gated
%   via \texttt{supabase.auth.getUser()}, Zod schema al request
%   (\texttt{content}, \texttt{availableTags}), límit de
%   20.000~chars al contingut, 500 etiquetes.
% - \textbf{Model}: \texttt{anthropic("claude-haiku-4-5")} amb
%   \texttt{generateObject()} del SDK. El schema de resposta és
%   \texttt{z.object(\{existing: z.array(z.string()), newTag:
%   z.string().nullable()\})}. La crítica de disseny inicial fou
%   afegir \texttt{.max(3)} al array, però Anthropic rebutja
%   \texttt{maxItems} al JSON Schema de structured output; el cap
%   de 3 es mou al post-processat (\texttt{.slice(0, 3)}) i es
%   reforça al prompt.
% - \textbf{Normalització}: l'etiqueta nova és coaccionada a
%   lowercase + kebab-case via regex
%   (\texttt{replace(/[\^{}a-z0-9-]/g, "-")} + colapsar guions) i
%   és refusada si col·lideix amb una etiqueta ja existent.
%
% \subsection{Client: hook debounced}
%
% \texttt{useTagSuggestions(content, availableTags, opts)} al
% fitxer \texttt{src/hooks/use-tag-suggestions.ts}. Característiques:
%
% - Debounce configurable (default 700~ms).
% - Minimum de caràcters (default 15) abans que un fetch
%   dispari.
% - \texttt{AbortController} per cancel·lar fetches obsolets quan
%   l'usuari segueix teclejant.
% - Mode \texttt{auto: false} per saltar-se el trigger automàtic
%   quan la nota ja té etiquetes (no té sentit re-thinking en
%   obrir una nota existent ja categoritzada).
% - Funció imperativa \texttt{trigger()} exposada per casos que
%   sí que volen una petició explícita (p.ex. l'usuari clica al
%   desplegable d'etiquetes en mode edit).
%
% \subsection{Lliçó defensable}

\section{Bug defensable: \texttt{search\_path} a funcions Postgres}
\label{sec:search-path-bug}
% Subsecció per al redactat final. Afegida 2026-04-24.
%
% Episodi d'enginyeria que il·lustra un patró de seguretat-vs-
% funcionalitat que apareix repetidament a la Part~B. La funció
% \texttt{public.get\_note\_graph()} va ser inicialment escrita
% amb \texttt{SET search\_path = ''} com a mesura de hardening
% contra schema hijacking (cap schema a la cerca $\Rightarrow$ cap
% possibilitat que un schema malicios del caller interfereixi).
%
% Efecte secundari: l'operador \texttt{<=>} de pgvector no es
% resolia, perquè pgvector viu al schema \texttt{public} i les
% cerques d'operador (a diferència de funcions i taules) no poden
% ser schema-qualificades inline. Producció retornava 500 amb:
%
% \begin{verbatim}
% operator does not exist: vector <=> vector
% \end{verbatim}
%
% \textbf{Primer intent de fix}:
% \texttt{SET search\_path = 'public, pg\_catalog'} (amb cometes
% al voltant de tota la llista). Error subtil: Postgres tracta
% aquest string com UN sol nom de schema literal anomenat
% \texttt{public, pg\_catalog}, que òbviament no existeix --- i
% la reexecució torna a fallar amb el mateix error.
%
% \textbf{Fix correcte}: \texttt{SET search\_path = public,
% pg\_catalog} (sense cometes, identifiers separats per coma).
%
% La lliçó aplicable a la Part~B: el hardening a nivell Postgres
% (search\_path restringit) és una defensa real contra certs
% vectors d'atac, però la sintaxi exacta importa i un error
% silenciós pot fer que la funció funcioni només en condicions
% específiques. El remei de disseny és verificar explícitament la
% configuració post-migració:
%
% \begin{verbatim}
% SELECT proconfig FROM pg_proc WHERE proname = 'get_note_graph';
% \end{verbatim}
%
% que hauria de retornar \texttt{search\_path=public, pg\_catalog}
% (sense cometes dobles al voltant de la llista).
